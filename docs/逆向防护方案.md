# 本地自动化「防逆向」分析与方案（存档）

> 结论先行：**现在不必动**（T0 君子锁 + 云端锁够用）；**等商业化收费时直接上 T1「AI 网关」**——
> 它不只是防逆向，而是「把账号变成产品命脉 + 保护提示词秘方 + 能计费」的商业基础设施，顺带把逆向问题一并解决。
> 配套：[权限架构.md](权限架构.md) §7 绕过边界。

## 1. 先想清楚：在保护什么？
真正值钱的**不是**编排代码（点击→抓取→回复，commodity，谁都能重写），而是：

| 资产 | 现状 | 风险方 |
|------|------|--------|
| 候选人**数据** | ✅ 已锁死（云端 RLS，服务端强制，绕不过）| — |
| **提示词**（system/scoring/intent prompt，= 回复质量&评分秘方）| ❌ 明文随包发出 | 抄袭者 |
| **AI 能力**（DeepSeek 调用）| ❌ 本地用本地 key 直连 | 白嫖额度者 |

→ 问题不是「怎么混淆 Python」，而是「**怎么让没账号的人用不了智能能力、也偷不到提示词**」。
→ **混淆编排逻辑 = 低 ROI**（可破 + 保护的是 commodity 代码 + 增加打包/调试痛苦）。

## 2. 是否必要：看阶段
- **现在（验证期 / 少量用户 / 熟人）**：当前「君子锁」(`require_account`) + 云端锁**够了**，过度工程拖累迭代。
- **开始收费 / 有竞品 / 盗版有真实成本时**：值得投入 T1。

## 3. 最佳方案：AI 网关（服务端代理）—— 一箭四雕
把 DeepSeek 调用 + 提示词搬到服务端（复用现有 Cloudflare Worker + Supabase 鉴权）：
```
本地 chat_reply/scoring ──带 user token──► AI 网关 Worker
                                            ① 校验账号有效(Supabase)
                                            ② 服务端持 DeepSeek key + 提示词
                                            ③ 调 DeepSeek → 返回结果
                                            ④ 计量 / 限流 / 可吊销
```
**四个收益**：
1. **账号变强制**：没有效账号 → 拿不到智能回复/评分 → 产品被掏空，改本地代码也救不回（服务端强制，真绕不过）。
2. **提示词永不出服务器**：秘方用户拿不到。
3. **成本可控 + 可计费**：DeepSeek 费用集中，按用户计量、可按调用收费。
4. **可远程吊销**：停账号 → AI 立即失效。

**代价**：多一跳延迟（国内→Cloudflare→DeepSeek）、你承担 DeepSeek 成本（需计费模型）、多一个服务组件 + 单点依赖。

## 4. 分层路线图（按阶段加，别一步到位）
| 层 | 做什么 | 何时 | ROI |
|----|--------|------|-----|
| **T0（现状）** | 君子锁 `require_account` + 云端 RLS 锁 | 已完成 | 够用 |
| **T1（收费时）** | **AI 网关**：提示词 + DeepSeek 调用服务端化，本地改走代理 + 计量 | 商业化 | ⭐最高 |
| **T2（盗版严重）** | + 代码混淆(Nuitka) + 每步关键操作 token 校验(可吊销) | 真有人盗 | 递减 |

## 5. T1 实施要点（将来动手时的清单）
- 新 Worker `bole-ai-proxy`：secret 持 `DEEPSEEK_API_KEY` + 提示词（从 `config/*.md` 迁入服务端）；endpoints `/ai/reply` `/ai/score` `/ai/intent`。
- 鉴权：复用 `bole-admin-api` 同款（校验 Supabase user token → 账号有效 → 放行）。
- 本地：`app/chat_reply.py` / `app/scoring.py` 增「网关模式」——有 `AI_GATEWAY_URL` 则走代理(带 user token)，否则回退本地 key（拥有者开发用）。
- 计量：Worker 记录每用户调用数 → 限流 / 配额 / 计费基础。
- 注意：延迟与失败需 best-effort 降级（网关挂 → 提示而非崩）；提示词版本管理。

## 6. 当前决定（2026-06-27）
**保持 T0，不额外加固**。理由：验证期盗版非主要矛盾；云端数据已固若金汤；本地编排是 commodity。
**触发 T1 的信号**：开始对外收费 / 出现抄袭或白嫖 / 需要按用量计费。届时按 §5 实施。

## 附录 A：逆向认知（为什么护城河只能在服务端）

> ⚖️ 逆向自己的软件 / 安全研究 / 互操作 / 恶意样本分析是正当的；破解他人商业软件去盗版/去授权违法。以下是方法论认知，用于「知己知彼」做防御决策。

### A.0 一条总铁律
> **代码要运行，最终必须以「能执行/能用的形态」出现在内存里。攻击者完全掌控运行它的机器（反汇编器/调试器/dump 内存），所以总能蹲在它「不得不现原形」的边界。** ⇒ **永远不要信任客户端。**

理论结论：**所有桌面程序都可被逆向；客户端的授权/登录校验几乎都可被绕**——区别只是难度/成本（分钟级 → 专家团队数月级），不是「能不能」。

### A.1 高门槛保护的原理与破解大致思路（认知向）
| 保护 | 原理 | 为什么终会被破（破解抓手）|
|------|------|--------------------------|
| **加壳** Packer | 原程序加密压缩 + 壳 stub；运行时把真代码**解密进内存**再跳 OEP | 真代码必须明文进内存 → 跑起来后从内存 **dump** + 重建导入表即脱壳 |
| **反调试** Anti-debug | 检测被调试（系统标志/时间差rdtsc/断点扫描/异常陷阱…）就退出或走假逻辑 | 检测本身是代码 → 定位后**打补丁让它永远返回「没被调试」**，或用隐身插件统一隐藏 |
| **VM 保护** 代码虚拟化 | 关键函数翻译成**自定义 VM 的私有字节码**，原始指令消失，只剩 VM 解释器 + 混淆 handler | ① 逆 VM 架构→识别 handler→反虚拟化还原逻辑；② 不啃内部，盯被保护函数的**输入/输出**与结果**流回普通代码**处（结果总得出 VM 才能用）|

标准流程：静态分析（反汇编/字符串/导入表）→ 动态分析（调试/内存/trace）→ 识别保护 → **逐层剥**（脱壳 → 关反调试 → 啃 VM/逻辑）→ 打补丁或提取。**核心是「攻最弱层 / 攻边界」，不硬刚最强部分。**

### A.2 这些手段的真实价值
- 加壳/反调试/VM **叠加**能把破解从分钟级推到月级（Denuvo 让部分游戏数月未破），**但历史上几乎都最终被攻破**。
- 它们保护的是**客户端逻辑**，而客户端永远在攻击者手里 → 只是「续命/买时间」，非「锁死」。

### A.3 唯一不可绕的防护
> **从未下发到客户端的东西，拿不走。** 把价值（逻辑/数据/密钥/算力）放服务端，客户端只是壳；壳被脱壳、反虚拟化、全破，也得不到服务器上的东西。
>
> 关键区分：**「破解了程序」≠「拿到了价值」**——价值在服务端，程序随便逆向也白费。

### A.4 对本项目的指导
- **别在桌面端搞军备竞赛**（投入大、终会破、拖累迭代）。VM 保护只适合「确实必须留在客户端、又极怕被抄的核心算法」——多数 SaaS 用不到。
- **护城河放服务端**：① 云端数据 RLS 服务端强制（✅ 已做）；② 收费时上 T1 AI 网关，把「没账号就用不了的能力（提示词+AI 调用）」搬服务端（§3/§5）。
- 本地 `require_account()` 定位清晰：**君子锁**——挡正常使用、不防逆向；真正的强约束在服务端那两条。
