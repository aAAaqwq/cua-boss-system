# 数据安全与采集说明

> 本文说明 cua-boss-system **采集什么数据、存在哪、怎么保护、谁能看**。
> 配套：[cloud-sync-plan.md](cloud-sync-plan.md)（云同步规划）、`supabase/schema.sql`（建表+RLS）。

---

## 1. 采集什么数据
通过本地 agent 操作 BOSS 直聘，采集**候选人**信息：
- 基本：姓名、学校、学历、沟通岗位
- 联系方式：手机、微信、邮箱（**敏感 PII**）
- 简历正文、简历文件名
- 沟通记录（聊天）、评分与评语、面试安排、状态

## 2. 存在哪里
| 位置 | 内容 | 默认 |
|------|------|------|
| **本地 SQLite** `data/candidates.db` | **全部数据（真源）** | 始终 |
| **本地文件** `data/resumes/` | 简历 PDF 原件 | 始终 |
| **本地备份** `data/backups/` | DB 备份 | 自动 |
| **云端 Supabase**（美区） | 全字段镜像（含 PII）| **仅 `CLOUD_SYNC=on` 时** |

> 本地是真源；云端是**可选**的下游镜像，关掉开关（`CLOUD_SYNC=off`）则完全不上云。

## 3. 鉴权与访问控制（谁能看云端数据）✅
**不是谁都能看。** 三层保护：
1. **必须登录**：网页用 Supabase Auth（邮箱+密码）认证，未登录拿不到任何数据。
2. **行级安全 RLS**：数据库规则 `tenant_id = auth.uid()` —— 登录后**只能看到自己 `tenant_id` 的行**，看不到别人的。
3. **实证**：用公开的 anon key 但**不登录**直接查 → 返回 **0 行**；账号 A 看到自己的全部、账号 B（空）看到 0 条。**多用户互相隔离。**

> anon key 放在前端是**设计如此、公开安全**的——它只是「入口钥匙」，真正的门禁是登录 + RLS。

### Agent 如何绑定账号（多用户标准做法）✅
**每个用户的本地 agent 必须「登录」自己的账号才能上云**——不发万能钥匙，登录后只能动自己的数据：
1. **账号由服务端创建/下发**（我们的系统注册或后台开通），用户拿到自己的**邮箱+密码**。
2. 用户在本地一次性绑定：
   ```
   python scripts/cloud_sync.py login --email 你的邮箱 --password 你的密码
   ```
   → agent 用账号密码换取 **user token**，存本地 `data/.cloud_auth.json`（权限 600、gitignored）。
3. 此后采集/沟通自动上云时，**以该账号身份**写入，**RLS 强制只能写自己 `tenant_id`**（= 登录账号 uid，自动绑定，无需手填）。
4. token 过期自动用 refresh_token 续期；`cloud_sync.py logout` 解绑；`--status` 看当前绑定账号。

> **关键安全点**：普通用户**只拿到自己的登录账号**，**绝不发 service_role**（那是绕过 RLS 的管理员钥匙，只在拥有者机器、仅作管理用）。「只有登录才能连接、且只能连自己的数据」由此保证。

## 4. 传输与存储保护
- **传输**：全程 HTTPS/TLS（到 Supabase、到 Cloudflare）。
- **静态加密**：Supabase 数据库默认开启磁盘静态加密。
- **本地密钥隔离**：所有密钥在 `.env`，已 `.gitignore`，**不进版本库**；`candidates.db`/`resumes/`/`cloud_queue.jsonl` 同样 gitignored。
- **AI 不能代发 PII**：平台硬性禁止 AI 智能体把候选人 PII 传出本机——上云这步**只能由你本人执行**（终端跑），从机制上防止代理误传。

## 5. 密钥管理
| 密钥 | 用途 | 位置 | 注意 |
|------|------|------|------|
| `DEEPSEEK_API_KEY` | AI 回复/评分 | `.env` | 勿外泄 |
| **user token**（登录换取） | **agent 上传（受 RLS，只能写自己）** | `data/.cloud_auth.json`(600) | 多用户标准路径；过期自动续期 |
| anon key | 登录入口 + 前端 | `.env`/前端(公开) | 公开安全，受 RLS 约束 |
| `SUPABASE_KEY`(service_role) | **仅拥有者管理**(绕 RLS) | `.env` | **绝密**，绝不发普通用户/前端/git |
| `SUPABASE_PAT` | 管理(建表/取key) | `.env` | **账号级**，**用完应 Revoke 轮换** |

## 6. 自动同步行为（CLOUD_SYNC=on）
- `cua_collect.py` / `cua_chat_loop.py` 在**本地存完后**，把本轮处理过的候选人**增量上传一份**。
- **best-effort**：未开/未配置→跳过；网络失败→入本地队列 `data/cloud_queue.jsonl`，下次自动补推；**任何云端异常都不影响本地采集与已入库数据**。
- 无 uid 的孤儿记录不上云（无法可靠去重）。
- 手动补全/全量：`python scripts/cloud_sync.py --push`；预览将上传什么：`--dry-run`；看状态：`--status`。

## 7. 合规与残留风险（务必知悉）
- **候选人未同意**：数据是从 BOSS 采集的，候选人未授权你存储/上云 → **PIPL/GDPR 法定义务**，免责声明盖不住「你↔候选人」。
- **数据控制者 = 你**：集中控管全量 PII → 合规责任在你；**拖库即大规模个人信息泄露**。
- **跨境**：Supabase 美区 → 数据出境（PIPL 关注点）。如需合规，考虑**境内自托管**。
- **BOSS ToS**：把候选人数据搬离平台存自有云可能违反其用户协议。
- **未做**：强 PII 列级加密（pgcrypto）尚未启用——目前靠磁盘静态加密 + RLS。
- **测试残留**：演示账号 `demo-a/b@bole-hr.test`（弱密码）验证完应删除；`SUPABASE_PAT` 应 Revoke。

## 9. 许可门禁 + 管理后台（账号管控）
- **账号只能后台创建**：Supabase 公开注册已**关闭**（`disable_signup=true`）；用户无法自助注册，账号**只能由拥有者用管理后台开通**。
- **必须登录才能驱动程序**：`boss_pipeline / cua_greeting_loop / cua_collect / cua_chat_loop` 启动时调 `cloud_sync.require_account()`——**未用我们下发的账号登录则直接拒绝运行**。停用某账号后，其 agent 续期时即失效。
- **管理后台**（`scripts/admin.py`，仅拥有者本机，用 service_role/PAT，绝不部署/进前端）：
  | 命令 | 作用 |
  |------|------|
  | `admin.py users` | 列出所有用户 + 各自数据量 + 状态 |
  | `admin.py create --email --password` | 开通账号(下发给用户) |
  | `admin.py disable/enable --email` | 停用/恢复账号 |
  | `admin.py data [--email]` | 各用户(或指定用户)数据概况 |
  > 这是 CLI 形态的后台；后续可做**网页版管理面板**（管理员账号 + RLS 管理策略，service_role 留服务端不进浏览器）。
